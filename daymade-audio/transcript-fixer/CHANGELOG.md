## [Unreleased]

### Added
- **Second-pass residual adjudication heuristics + trap-scan (Native AI Correction step 6, doc-only).** The cold subagent's residual list is now explicitly *candidates, not conclusions* (one real run: 10 rows → 4 accepted), with six production-validated accept/reject rules: near-homophone + in-document self-proof accepts; sound-distance falsifies implausible swaps (the sound test now cuts both ways); the **ASR-capability counter-check** (a word the same engine rendered correctly nearby is inside its recognition range, so a different rendering is more likely what was said — a strong prior, not a proof); intelligible real words stay; evidence-free reconstructions die; minimal edit only (never insert unspoken words). New fast-tier **trap-scan**: grep the file for the domain context file's documented trap patterns before (or instead of) a full cold re-read.
- **User-verdict compounding loop (step 4, doc-only).** A user's mid-session resolution of an uncertain item is now treated as the strongest source and cashed in three ways immediately: apply the fix → persist the variant (roster for recurring people, `--add --domain` for project terms) → record it in the ledger/roster/context with verbatim wording, date, and a ✅ user-confirmed marker. Two refinements: collision-check the FROM side against real people in other projects before dict-adding (real-name → real-name fixes go to the domain context as a trap, never the dictionary); confirmed-*correct* entities get a one-line note too, so the next run doesn't re-ask. Dictionary decision matrix gains a "real name → different real name: never a rule" row.
- **`--enqueue-review` item schema documented** (`references/script_parameters.md` §Review Queue Item Schema): full field/alias table (only `original` required), the silent-drop gotcha for unknown keys (`line`, not `line_hint`), temp-dir items skipped entirely, CLI `--domain` fills only items without their own, empty-`suggested` + no-action-pack errors at resolve time, and the five-field dedup key. SKILL.md step 7 cross-references it.
- **Persistent review queue** (`review_items` table): uncertain corrections now have a durable home instead of evaporating. Unifies three previously-lossy flows — native AI-pass uncertain items (were chat-only), Stage 1 safe-mode deferrals (were a `*_needs_review.md` sidecar that temp-dir callers silently discarded; deferrals now auto-enqueue, with temp-dir anchors skipped as dead pointers), and learned-suggestion review. CLI verbs: `--enqueue-review` / `--list-review` / `--show-review` / `--resolve-review ID --decision …`. Each item carries a pre-filled suggestion, evidence, and an optional action pack (`file_edit`/`dict_add`/`append_note`) executed on accept via an in-memory two-phase planner: every action validates against the file state the pack's previous actions produced, and nothing reaches disk unless the whole pack plans clean. Anchor selection is content-first (context snippet outranks a stale line hint), ambiguity fails closed (`re_anchor_needed`, exit 2, machine-readable on stdout), file IO preserves CRLF/LF, and resolve/reopen use conditional-claim UPDATEs so two writers can't silently overwrite each other's verdict. `--json` Stage 1 status gains an additive `review_enqueued` field.
- **Review dashboard** (`scripts/review-dashboard/server.py`, local FastAPI at `127.0.0.1:8767`): Prodigy-style one-keystroke review of the queue — `A` accept / `R` original-is-correct / `W` override / `S` skip / `Z` undo (reopen) / `Q` play the anchored utterance / `↑↓ J K` navigate. Reads the DB read-only; every write shells out to the CLI so the state machine and audit log stay the single source of truth. Audio playback is opt-in per transcript via an explicit frontmatter `audio:` field (unquoted, `~`-expanded, relative paths resolved against the transcript's directory, `is_file()` gated — no ghost buttons) with the clip window derived from surrounding speaker timestamps and served with HTTP Range. Env: `REVIEW_DASHBOARD_PORT`, `REVIEW_DASHBOARD_NO_BROWSER`. Hardened by a status×feature test matrix plus adversarial-probe regressions (38 tests in `test_review_queue.py` + `test_dashboard_context.py`): decided items keep context/audio by anchoring on the resolved text, stale context responses can't bind audio to the wrong card, background refresh never repaints while typing or listening.
- **Domain correction contexts (per-domain AI priors).** New convention: one markdown file per domain at `~/.transcript-fixer/contexts/<domain>.md` (user space, survives skill updates), read by the native AI-correction workflow before triage. It carries the third error class that fits neither the dictionary nor the people roster: **context-dependent homophones** (e.g. `减`→`剪` only when the discussion is about producing N video clips) — as *cues that prime judgment*, never as hard replacement rules. SKILL.md documents the convention (new "Domain Correction Contexts" section, workflow step 3, dictionary decision matrix now routes common-word homophones here); format + worked template in `references/domain_context_guide.md`. Native-only for now: API mode (`--stage 2/3`, backup channel) does not inject contexts yet.

### Fixed
- **`--apply-all` no longer gets hijacked by a stale `_stage1.md`.** The auto-finalize promote guard ran before corrections for any stage-1 invocation: if a previous safe-mode run had left `_stage1.md` behind (possibly with ZERO corrections applied — safe mode defers medium/high rules), an explicit `--apply-all` promoted that stale sidecar, printed "Finalize complete", and exited — the requested correction run never happened and the ASR errors stayed in the file (observed 2026-07-10: 13 high-risk name fixes silently skipped on a real meeting transcript). `--apply-all` now skips the promote path and always runs corrections from the input as-is; finalize remains the job of a plain `--stage 1` re-run. Regression test: `scripts/tests/test_stage1_auto_finalize.py::test_cmd_apply_all_ignores_stale_stage1_and_runs_corrections`.
- **Stage 3's diff report no longer reads a `_stage1.md` this run didn't produce.** On a 0-correction run `_stage1.md` is skipped (see below), but the Stage 3 report still passed that path to `generate_full_report`: with no leftover file the report failed with a swallowed `FileNotFoundError`; with a stale leftover from a previous run it silently mixed old content into the diff. The report now reads the input file as the stage-1 column whenever this run applied 0 corrections (byte-identical by definition).
- **Native/stage-1 runs no longer nag about a missing API key.** `get_config()` logged `Configuration warnings: ['API key not set (required for AI corrections)']` on every command, including the default native workflow that never touches the API (the key has been unset for 8.5 months of daily native use). The warning is now gated behind `Config.validate(include_api_key_warning=...)`: suppressed at generic config load, still surfaced by the explicit `--validate` health check, and `--stage 2/3` keeps its own fail-fast error.
- **`--output FILE.md` no longer creates a directory of that name (false-success footgun).** `--output` was always treated as a directory: passing a file path (e.g. `--output result.md`) silently ran `mkdir result.md/` and wrote `<stem>_stage1.md` inside it, while the console printed only a basename (`Saved: <stem>_stage1.md`) — the run looked successful but the file was not where it was asked for. A `--output` whose suffix is `.md`/`.markdown`/`.txt` and is not an existing directory is now written directly to that exact path (directory `--output` unchanged, backward-compatible). Every `Saved` / changes-report / needs-review / dry-run / stage-2 line now prints the **full resolved path** instead of a bare basename, so a misdirected output is immediately visible.
- **`--add` on a soft-deleted rule no longer crashes with a misleading "Correction not found".** When `(from_text, domain)` already existed but was disabled (`is_active=0`, e.g. previously reported as a false positive via `--report-false-positive`), the UNIQUE-conflict handler updated only `is_active=1` rows, matched nothing, and raised `Correction not found` — a crash, not a message. `add_correction` now queries the existing row regardless of `is_active` and upserts deliberately: an active row is updated in place; a disabled row is **not** silently resurrected — without `--force` it reports the disable reason/date + old target and instructs to re-run with `--force`, and with `--force` it reactivates and updates the target, logging a `reactivate_correction` audit entry. `force` is threaded CLI → service → repository. Documented in `references/false_positive_guide.md` (§Re-adding a disabled rule).

### Changed
- **Stage 1 re-runs now auto-finalize**: if `*_stage1.md` exists and is newer than the input file, running `--stage 1` again atomically promotes it to the input filename and removes intermediate sidecars (`_stage2.md`, `_dryrun.md`, `_changes.md`, `_needs_review.md`, `_uncertain.md`, `_对比.html`). It skips promotion when the input file is newer, so manual edits are never overwritten. This removes the need for a separate manual finalize command in the native AI-correction workflow.
- **Stage 1 no-op runs write no sidecars.** When a `--stage 1` run applies 0 corrections (clean transcript, or a native re-run after the input was already edited), it no longer writes `_stage1.md` (a byte-copy of the input); when nothing was deferred either, `_changes.md` ("No corrections applied") is skipped too. Both were pure noise: they never auto-finalized (the promote guard skips when the input is newer — exactly the native case) and forced a manual `rm`. When safe mode does defer medium/high rules, `_changes.md` and `_needs_review.md` still write — they are the deferral report, so human review is never lost. Regression test in `scripts/tests/test_stage1_auto_finalize.py::test_cmd_stage1_zero_corrections_writes_no_sidecars`.
- Auto-finalize promotion uses `os.replace` with a `shutil.copy2` fallback so it works even when the input file and sidecars live on different filesystems.
- Learned-suggestion review is now wired end-to-end: `--review-learned` analyzes SQLite `correction_history` / `correction_changes`, merges high-confidence repeated patterns into `~/.transcript-fixer/learned/pending_review.json` without duplicates, and `--approve FROM TO` adds the exact suggestion to the dictionary as `source='learned'` before removing all matching pending rows.
- Correction dictionary import/export is now exposed in the CLI: `--export PATH` writes a metadata-wrapped JSON export for the selected `--domain`, and `--import PATH [--merge]` reads both the current wrapped format and legacy plain `{from: to}` JSON dictionaries.
- **Stage 1 now defaults to "safe mode"**: only low-risk (non-word, high-confidence) corrections are auto-applied. Medium/high-risk ones (common words, ≤2-char, real-word fragments) are tracked to `*_needs_review.md` for human/AI review instead of being applied silently. `Applied: 0` on a clean transcript is now expected, not a bug. Pass `--apply-all` to restore the previous apply-every-level behavior.
- Expanded `utils/common_words.py` with real common words that had been mis-added as correction sources (`多深`, `早生`, `龙虾`, `小龙虾`), so the add-time guard, the risk classifier, and `--audit` all recognize them.
- Context rules are now gated by `review_mode` like dictionary rules (safe mode defers risky ones instead of applying them unconditionally).
- `--audit` runs an advisory jieba heuristic to surface 4+ char "real-word" false-positive rules for human review (see Known limitations). Adds `jieba` to dependencies.
- **Finalize now prescribes `/bin/mv -f` instead of a bare `mv`.** SKILL.md step 9 and new troubleshooting entry #7: on macOS `mv` is commonly aliased to `mv -i`, which skips overwriting an existing target while still exiting 0 — so a finalize `mv *_stage1.md *.md && echo done` reported success while the un-corrected file silently survived as the output. Doc-only change.
- **Native AI Correction step-4 ("Needs verification") now prescribes a local-first search ladder.** Previously the instruction was "Search it, don't guess — WebSearch, or a local grep," which put WebSearch first and left "local grep" vague — so project / person names whose canonical spelling already lived in another `corrections.db` domain or in project delivery docs got escalated to the user instead of resolved locally. The ladder now orders: all `corrections.db` domains (not just the current `--domain`) → project delivery docs → memory → WebSearch (public entities only) → then the user. Doc-only.

### Deprecated
- `--review` is now a no-op (safe mode is the default). Use `--apply-all` for the opposite behavior.

### Fixed
- **False-positive class where the risk guard was computed but then ignored.** `_assess_risk()` already classified risky rules correctly (`多深`→high, `小龙虾`→medium), but `review_mode` defaulted to `False`, so every risk level was applied regardless. On a clean Feishu-ASR transcript this silently corrupted correct text (`抓多深`→`抓多申`, `小龙虾`→`小 Claude`). Safe-mode-by-default now defers all medium/high-risk changes — dictionary **and** context rules. This narrows the class but does NOT fully close it (see Known limitations). Regression tests in `tests/test_common_words_safety.py::TestProductionFalsePositives2026_06`.
- **`_apply_context_rules` now honors `review_mode`.** Previously context (regex) rules were applied unconditionally even in safe mode, and the run summary mis-counted them as "skipped" while the text was already mutated. They are now risk-gated like dictionary rules, so the summary and `*_needs_review.md` are accurate.
- **`fix_transcript_enhanced.py` no longer silently flips to safe mode.** Its hand-built args now pass `apply_all=True`, preserving this automation entry point's historical "apply everything" behavior (the safe-mode default had silently downgraded it with no opt-out).
- **History no longer records non-applied changes.** In safe mode, skipped medium/high Stage-1 changes were persisted to the history table as if applied; only actually-applied changes are now recorded.
- **Fixed SQLite connection pool thread-safety bug.** `ConnectionPool` was creating connections with `check_same_thread=True`, then handing them to worker threads via a queue. SQLite forbids using a connection outside its creation thread, so concurrent tests failed with "SQLite objects created in a thread can only be used in that same thread." Switched to `check_same_thread=False` (safe because the queue enforces one-thread-at-a-time exclusive access).
- **Fixed test suite import and dependency issues.** Added missing `sys.path` setup to `scripts/tests/test_connection_pool.py`, made `test_apply_all_applies_real_word_phrase` skip when `jieba` is unavailable, and added `pytest` / `pytest-asyncio` to `requirements.txt` so the existing async/error-recovery tests can run.

### Known limitations
- **Safe mode does NOT catch the "4+ char real-word" false-positive class.** `_assess_risk` labels any rule with `len(from_text) >= 4`, confidence ≥ 0.9, and not in the common-word list as `low`, so it auto-applies — even when `from_text` is itself valid text (`济南大学`→`暨南大学`, `关税证明`→`完税证明`). This is structural: "low risk" is defined by length/confidence/word-list, which is orthogonal to "is this real text." `--audit` now uses a jieba heuristic (`is_likely_valid_phrase`) to **surface** such rules for human review, but it is advisory and low-precision (it also flags many legitimate ASR-garble rules, not just genuine false positives), so it cannot gate auto-application. Fully closing this class needs language-model-grade validity judgment, not a dictionary heuristic.

### Security
- API keys are now loaded from the canonical config directory (`~/.transcript-fixer/config.json`) first. Environment variables (`GLM_API_KEY`, `ANTHROPIC_API_KEY`, `ANTHROPIC_BASE_URL`) are treated as explicit overrides only.
- Removed all shell-config-file scraping for secrets.
- Config directory restricted to `0o700` and config file to `0o600`.

### Added
- `scripts/core/defaults.py` — single source of truth for all variable defaults: AI provider/model/base URL/auth header, timeouts, chunk size, file permissions, and database `system_config` defaults.
- Database `system_config` defaults are now written by `CorrectionRepository._initialize_system_config()` from `core.defaults.SYSTEM_CONFIG_DEFAULTS` instead of being hard-coded in `schema.sql`.
- `--validate` now checks that `system_config.api_provider`, `api_model`, and `api_base_url` match the canonical `core.defaults` values and reports drift as an error.
- Added migration `v2.3` to align existing databases with the new canonical defaults.

### Changed

- Updated default AI models to `GLM-5.2` (primary) and `GLM-5-turbo` (fallback).
- Deduplicated sync/async AI processor logic into `scripts/core/ai_utils.py` (single source of truth for chunking, prompt building, and response parsing).
- `fix_transcript_enhanced.py` now calls the correction pipeline directly instead of spawning a `subprocess` wrapper.
- `httpx` clients use `http2=False` to avoid the missing `h2` package.
- `scripts/core/ai_processor.py` and `scripts/core/ai_processor_async.py` now import all AI defaults from `core.defaults`.
- `scripts/utils/config.py` now imports app name/version, timeout, retries, and permission modes from `core.defaults`.
- `scripts/utils/migrations.py` now interpolates default model/provider/base URL/domain from `core.defaults` instead of hard-coding them.
- `scripts/utils/diff_formats/markdown_format.py` now takes an optional `model` parameter and defaults to `DEFAULT_MODEL` instead of a hard-coded model name.
